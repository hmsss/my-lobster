## 标准工作流

```
老板下达任务
↓
CEO 接收 → 理解需求 → 拆解分配
↓
叙事架构师 执行（自我播报）→ CEO 审核
    ├─ 不通过 → 打回重做 → 叙事架构师播报修改进度
    └─ 通过
↓
写手 执行（自我播报）→ CEO 最终审核
    ├─ 不通过 → ❗回退叙事架构师重做
    └─ 通过 → 任务完成
↓
CEO 向老板汇报
```

## ⚠️ CEO 回复规范（防止重复通知）

### sub-agent 完成后 CEO 的正确处理流程

当收到 sub-agent 完成事件时：
1. **不要立即回复用户** — sub-agent 已经直接通知过用户了
2. **先审核产出** — 读取文件，检查内容
3. **以 CEO 身份发送自己的消息** — 内容要包含 CEO 审核意见，不是简单转发

### 禁止事项

❌ **禁止这样回复**：
> "内容策划已完成题材提案，共8个题材供选择，请CEO分发老板。"

（这是 sub-agent 的通知，不应该原文转发给用户）

✅ **正确做法**：

收到 sub-agent 完成事件后，CEO 发送自己的消息：
```
【CEO（总指挥）审核结果】

内容策划已完成题材提案，共8个供选择：

1. 都市情感：...
2. 悬疑推理：...
...

请老板选择一个题材。
```

关键区别：
- sub-agent 发的是"任务完成通知"
- CEO 发的是"CEO 审核+分发老板选择"

两者内容不同，不会重复。

## 强制规则

1. **不允许静默** — 每步必须自我播报
2. **CEO 必须给结构化反馈** — 问题点 + 修改建议
3. **所有失败必须回到叙事架构师** — 写手不修 Bug，叙事架构师负责兜底
4. **输出必须可交付**
5. **CEO 不转发 sub-agent 的通知，要发自己格式的审核消息**
