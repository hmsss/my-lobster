## 1. 背景与问题陈述

### 为什么做这个产品？
中国有丰富的方言文化，但缺乏一个便捷的工具让用户探索和聆听各地方言。同时，用户也需要一个简单的音乐播放工具。

### 用户痛点
1. **方言爱好者**：想了解不同地区的方言发音，但找不到集中的资源
2. **文化研究者**：需要一个工具来对比不同地区的方言特点
3. **普通用户**：想要一个简洁的音乐播放器

### 目标用户
- 对中国方言文化感兴趣的用户
- 需要简单音乐播放功能的用户
- 语言学习者和文化研究者

## 3. Non-Goals（不做的事）

明确说明本次迭代 **不会涉及** 的内容：

| 不做的事 | 原因 | 重新考虑条件 |
|----------|------|--------------|
| 用户注册/登录系统 | V1 聚焦核心播放功能，无云端同步需求 | 用户反馈需要跨设备同步 |
| 方言录制/上传功能 | 需要审核机制，复杂度高 | V1 验证用户需求后迭代 |
| 移动端 App | 老板要求 Web 版本，移动端用响应式适配 | 用户数据显示移动端使用 >30% |
| 付费会员/VIP | 需要先验证产品价值 | 日活 > 1000 时考虑 |
| 社交分享功能 | 非核心功能 | 用户反馈强烈时迭代 |
| 方言文字展示 | 增加数据准备成本 | V2 考虑 |
| 离线播放 | 技术复杂度高 | 用户反馈网络问题频繁时 |

## 4. 用户故事

### 故事 1：浏览方言
**作为** 一名方言爱好者
**我希望** 能通过选择省市来浏览不同地区的方言
**以便** 了解各地方言的特点

**验收标准**：
- [ ] 首页展示省市级联选择器
- [ ] 选择省份后，显示该省份的城市列表
- [ ] 选择城市后，显示该地区的方言音频列表
- [ ] 列表展示方言名称、时长、简介

### 故事 2：播放方言
**作为** 一名用户
**我希望** 点击方言条目即可播放
**以便** 聆听该地区的方言发音

**验收标准**：
- [ ] 点击播放按钮开始播放
- [ ] 支持暂停、继续播放
- [ ] 显示播放进度条，可拖动调整进度
- [ ] 显示当前播放时长和总时长
- [ ] 支持音量调节

### 故事 3：播放音乐
**作为** 一名用户
**我希望** 能播放音乐文件
**以便** 欣赏音乐

**验收标准**：
- [ ] 首页提供"音乐"和"方言"两个标签页切换
- [ ] 音乐模块支持本地上传音乐文件
- [ ] 支持播放、暂停、上一首、下一首
- [ ] 显示播放列表
- [ ] 支持进度控制和音量调节

### 故事 4：搜索功能（P1 - 建议有）
**作为** 一名用户
**我希望** 能搜索方言或音乐
**以便** 快速找到我想听的内容

**验收标准**：
- [ ] 提供搜索框
- [ ] 支持按地区名称搜索方言
- [ ] 支持按音乐名称搜索（音乐模块）

## 6. 交互设计

### 页面结构
```
┌─────────────────────────────────────┐
│  Logo     [方言] [音乐]      🔍搜索  │  <- 顶部导航
├─────────────────────────────────────┤
│                                     │
│  ┌─────────┐  ┌─────────────────┐  │
│  │ 省份    │  │                 │  │
│  │ 列表    │  │   方言列表      │  │
│  │         │  │   (名称/时长)   │  │
│  │ 城市    │  │                 │  │
│  │ 列表    │  │                 │  │
│  └─────────┘  └─────────────────┘  │
│                                     │
├─────────────────────────────────────┤
│  ▶ ━━━━━━━━━━━○━━━━━  🔊 00:30/02:00│  <- 底部播放器
│  正在播放：四川成都话 - 日常对话     │
└─────────────────────────────────────┘
```

### 关键交互
1. **省市选择**：左侧树形选择器，点击省份展开城市
2. **播放操作**：列表项点击播放，底部固定播放器
3. **标签切换**：顶部标签切换，保持播放器状态

## 7. 接口需求

### 6.1 方言模块接口

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/dialect/provinces` | GET | 获取省份列表 |
| `/api/dialect/cities?province={id}` | GET | 获取城市列表 |
| `/api/dialect/list?city={id}` | GET | 获取方言音频列表 |
| `/api/dialect/audio/{id}` | GET | 获取方言音频文件 |

**响应示例**：
```json
// GET /api/dialect/list?city=chengdu
{
  "code": 0,
  "data": [
    {
      "id": "d001",
      "name": "成都话 - 日常对话",
      "duration": 120,
      "description": "成都本地人的日常对话录音",
      "audioUrl": "/audio/dialect/d001.mp3"
    }
  ]
}
```

### 6.2 音乐模块接口

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/music/upload` | POST | 上传音乐文件 |
| `/api/music/list` | GET | 获取音乐列表 |
| `/api/music/{id}` | DELETE | 删除音乐 |

## 8. 数据需求

### 7.1 方言音频数据（关键问题）

**方案一：第三方 API**
- 优点：数据现成，开发快
- 缺点：可能有版权问题，API 稳定性依赖第三方
- 可行性：需要调研是否有可用的方言音频 API

**方案二：自建数据**
- 优点：数据可控，无版权风险
- 缺点：需要采集/录制，周期长
- 建议：V1 版本先准备少量示例数据（31省各1-2条）

**方案三：开源数据集**
- 优点：免费，合规
- 缺点：需要整理格式
- 调研：搜索开源方言语音数据集

**建议**：
- V1 版本采用方案二 + 方案三混合
- 准备 31 省各 1-2 条示例方言音频
- 后续逐步补充更多内容

### 7.2 地区数据
- 使用标准省市区数据 JSON
- 来源：国家统计局公开数据
