## 3. 缺陷列表

### BUG-001：日期校验失效-未来日期可提交
| 项目 | 内容 |
|------|------|
| **严重程度** | 🔴 高 |
| **优先级** | P0 |
| **模块** | 工时模块 |
| **接口** | POST /api/timesheet |
| **描述** | 可以提交未来日期的工时记录，违反业务规则"不能超过今天" |
| **复现步骤** | 1. 登录获取 Token<br>2. POST /api/timesheet<br>3. date 参数填写未来日期（如 2026-03-30） |
| **预期结果** | code=2002，提示"日期超出允许范围" |
| **实际结果** | code=0，提交成功 |
| **影响** | 员工可以预先填写未来工时，影响工时数据真实性 |
| **建议** | 后端增加日期校验：`date <= 今天` |

### BUG-002：日期校验失效-超过7天前可提交
| 项目 | 内容 |
|------|------|
| **严重程度** | 🔴 高 |
| **优先级** | P0 |
| **模块** | 工时模块 |
| **接口** | POST /api/timesheet |
| **描述** | 可以提交超过7天前的工时记录，违反业务规则"不能早于7天前" |
| **复现步骤** | 1. 登录获取 Token<br>2. POST /api/timesheet<br>3. date 参数填写10天前日期（如 2026-03-01） |
| **预期结果** | code=2002，提示"日期超出允许范围" |
| **实际结果** | code=0，提交成功 |
| **影响** | 员工可以补填很久之前的工时，影响工时数据准确性 |
| **建议** | 后端增加日期校验：`date >= 今天 - 7天` |

### BUG-003：未认证响应格式不一致
| 项目 | 内容 |
|------|------|
| **严重程度** | 🟡 中 |
| **优先级** | P1 |
| **模块** | 认证模块 |
| **接口** | GET /api/auth/me, POST /api/timesheet 等 |
| **描述** | 未携带 Token 时，返回 FastAPI 默认格式 `{"detail":"Not authenticated"}`，而非接口文档定义的 `{"code":1002,"message":"未授权"}` |
| **复现步骤** | GET /api/auth/me 不带 Authorization 头 |
| **预期结果** | `{"code":1002,"message":"未授权"}` |
| **实际结果** | `{"detail":"Not authenticated"}` |
| **影响** | 前端错误处理逻辑可能失效 |
| **建议** | 统一使用自定义异常处理器，返回标准格式 |

## 5. 测试结论

### 5.1 总体评估
| 评估项 | 结论 |
|--------|------|
| **是否可上线** | ❌ **否** |
| **风险等级** | 🔴 高风险 |
| **建议** | 修复 BUG-001、BUG-002 后重新测试 |

### 5.2 风险分析

1. **高风险**：日期校验完全失效（BUG-001、002）
   - 员工可以随意填写任意日期的工时
   - 可能导致工时数据混乱、统计失真
   - 必须在上线前修复

2. **中风险**：响应格式不一致（BUG-003）
   - 不影响功能使用，但影响前端错误处理
   - 建议在后续迭代修复

### 5.3 下一步建议

1. **立即修复**：
   - 在后端 `timesheet` 路由中添加日期校验逻辑
   - 校验规则：`今天 - 7天 <= date <= 今天`

2. **修复后回归**：
   - 重新执行 TC-TS-004、TC-TS-005
   - 确认返回 code=2002

3. **可选优化**：
   - 统一未认证响应格式
   - 补充前端日期选择器的禁用逻辑
